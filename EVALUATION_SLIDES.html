タップ/スワイプで進む

Itera OS v2 (itera2-dev)
設計評価まとめ

  • 評価日: 2026-07-19
  • 評価者: Itera (秘書AI)
  • 観点: アーキテクチャ / VFS / AIループ / プロセス / モバイル

総評

  • 「OSを構築するOS」としての模範的モジュール設計
  • 責務分離が徹底 (core / shell / vfs / ipc)
  • モバイル運用で顕在化する依存・ハードコード課題
  • テスト基盤欠如・TS検査対象外のVFS_root
  • 総合スコア: 8 / 10 (改善余地は明確)

アーキテクチャ観点

  • 層分離: cognitive / control / vfs / state / ipc
  • Vite相対パス出力でポータブル
  • strict TS設定 (noUnusedLocals等)
  • vfs_root内のHTML/JSが型チェック除外
  • テストフレームワーク未導入

VFS観点

  • Facade + Operationパターン、ACL/トランザクション
  • ProviderManagerでクラウド同期抽象化
  • パス文字列での永久削除判定 (ad-hoc)
  • lock+retryのコピペ boilarplate
  • JSON.parse(JSON.stringify)クローン蔓延

AIループ観点

  • Max連続ツールguard・Abort安全設計
  • debounceでping集約・起床判定分離
  • content文字列includesで分岐 (system_task)
  • setTimeout順次実行のad-hoc並列制御
  • localStorage直接参照でシークレット散在

プロセス観点

  • RpcManagerのタイムアウト管理が堅牢
  • GuestCompilerの循環参照防止
  • home.htmlだけ固定PID特例
  • DOMボタン直接結合 (責務混在)
  • スクリーンショットで外部CDN注入 (無効)

モバイル適性

  • CDN依存でオフライン時に機能喪失
  • 多Manager/Operationでメモリ圧迫リスク
  • viewport基準の設計だが実機413px未検証
  • take_screenshotが現在利用不可 (CDN起因)

改善優先度

  • 1. スクリーンショット庫のVFS内包 (CDN削除)
  • 2. VFSポリシー interface抽出 (パス判定)
  • 3. BaseOperationへretryヘルパー集約
  • 4. テスト導入 (Vitest)
  • 5. TS include を vfs_root へ拡張

結論

  • 設計思想は非常に優秀・拡張性高い
  • 実用上の壁は「モバイル・オフライン」と「アドホック特例」
  • 上記優先度を潰せば製品級のOS基盤へ
  • 次ステップ: リファクタチケット化を提案可能
